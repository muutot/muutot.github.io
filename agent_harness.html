Agent Harness：智能体驾驭系统

—— 深刻改变 AI 工程面貌的隐形基础设施

今天我想和大家聊一个正在深刻改变 AI 工程面貌的概念——Agent Harness，中文可以叫"智能体驾驭系统"。

2026 年春天，这个词突然火了起来。但真正点燃讨论的，是一组让人坐不住的数字：

这些案例指向同一个发现：当模型能力跨过某个阈值后，限制因素不再是模型本身，而是包裹在模型外的那层基础设施。

一、Harness 是什么？

简单说，Harness 就是大模型之外的全部工程化基础设施。

“如果你不是模型，你就是 Harness。” —— LangChain 工程师 Vivek Trivedi

从系统提示词、工具调用到状态管理、安全沙箱，所有模型之外的代码、配置与执行逻辑，都属于 Harness 的范畴。我们可以用建筑行业的脚手架（Scaffolding）来理解它：没有它，工人无法到达上层楼层；有了它，工人可以触及原本够不到的高度。最关键的是，建筑完工时脚手架会被拆除。

二、生产级 Harness 的十二个核心组件

综合 Anthropic、OpenAI、LangChain 等前沿社区的实践，一个生产级 Harness 包含以下 12 个独立组件：

1 编排循环 —— Agent 的心跳

管理轮次，处理工具调用失败、检测死循环并响应安全边界。如 Anthropic 开发的 "Ralph Loop" 模式，通过文件系统提供跨上下文窗口的连续性。

2 工具系统 —— Agent 的双手

生产实践揭示了一个反直觉的原则：更多工具通常意味着更差的性能。工具过多会导致选择困惑。安全上须采用操作系统级沙箱（如 Linux 的 bwrap）进行进程隔离。

3 记忆系统 —— 跨越时间的状态保持

建立三层记忆架构：上下文内记忆（最快但最脆）、外部文件记忆（memory.md 指针索引）和项目级静态记忆（CLAUDE.md）。核心原则是：Agent 不信任自己的记忆，行动前需验证。

4 上下文管理 —— 在噪音中找到信号

应对“上下文腐烂”问题（关键内容在中间时性能下降超30%）。通常采用快速剪枝、微压缩、渐进坍塌、自动压缩和反应式压缩五种策略。

5 Prompt 构建 —— 模型实际看到的世界

告别“泥球架构”，采用多层组装模型（基础人格层、角色指令层、工具定义层、上下文注入层、动态规则层），配合严格的优先级栈防止提示词注入。

6 输出解析 —— 从自由文本到结构化行动

演进至第三代：Grammar-Constrained Decoding（语法约束解码）。通过在生成时数学化限制 Token 概率，实现接近 100% 的 Schema 合规率。

7 状态管理 —— 让任务可恢复、可调试

将状态建模为可溯源的流。具备“时间旅行调试（Time-Travel Debugging）”能力与“中断后恢复”能力，或像 Claude Code 一样利用 Git 提交天然作为检查点。

8 错误处理 —— 在必然的错误中生存

在长流程中错误是必然的。系统需要区分瞬态错误、LLM可恢复错误、用户可修复错误和意外错误。绝不因为单个工具调用失败而终止循环，而是包装为模型可理解的格式返回。

9 安全防护 —— 速度与安全之间的永恒张力

构建输入防护（Tripwire 熔断）、工具防护（参数验证）与输出防护（PII检测）三层体系。或采用 Stripe 的“宽容模式”——在完全隔离的沙箱环境里给 Agent 最大自由。

10 验证循环 —— 从 Demo 到生产的分水岭

包含三种互补验证：基于规则的反馈（测试套件/Linter）、视觉反馈（Playwright 截图验证 UI）以及 LLM-as-Judge（子 Agent 评估）。

11 子智能体编排 —— 并行化的艺术

任务过载时支持 Fork（上下文副本）、Teammate（面板协作）或 Worktree（独立分支）三种执行模型。或采用 Handoffs（交接模式）与 Agents-as-Tools（工具模式）。

12 终止条件 —— 知道何时停下

实现分层终止：自然终止、轮次限制、Token预算耗尽、安全拒绝等。解决 Agent 在核心任务完成后仍无限循环、无效重构代码的“完成问题”。

三、七个关键设计决策

每个 Harness 架构师在面对复杂系统时，必须做出的七个抉择：

1. 单 Agent 还是多 Agent： 优先榨干单 Agent 能力，工具超过 10 个或领域完全分离时再考虑拆分。
2. ReAct 还是 Plan-and-Execute： 前者灵活应对高确定性任务，后者能带来数倍的速度提升。
3. 上下文窗口管理策略： 优先保留推理痕迹而非原始工具输出，能在保持准确率的同时减少 26%~54% 的 Token 消耗。
4. 验证循环设计： 结合计算验证（提供确定性真相）与推理验证（捕捉语义问题）。
5. 权限与安全架构： 在极限制约模式（安全缓慢）与完全沙箱化宽容模式（快速高效）中寻找平衡。
6. 工具范围策略： 严格奉行最小暴露原则，只赋予当前步骤所需的最小工具集。
7. Harness 厚度： 2026年最大的架构押注。你更倾向于把控制权给明式图框架（厚 Harness），还是寄希望于模型的飞速迭代（薄 Harness）？

四、社区的核心争论

关于 Harness 的“厚度”，社区至今分成明显的两派：

模型派 认为 Harness 被过度炒作了。Claude Code 创造者 Boris Cherny 坦言：“所有秘诀都在模型本身，它只是模型上最薄的一层包装。” OpenAI 的 Noam Brown 也指出，在新模型面前，大费周章搭建的复杂编排逻辑很容易变成明天的绊脚石。

Harness 派 则坚信基础设施决定了上限。LangChain 的 Jerry Liu 认为，AI 价值的真正障碍在于用户工程化工作流的能力。TerminalBench 同模型不同排名（相差25位）的铁证，就是 Harness 决定性差距的最好体现。

Latent Space 社区提出了精妙的“脚手架原则”：为移除而构建。Harness 的最终设计目标，应当是随着模型自身能力的飞跃，而主动降低其自身的复杂性。

五、结论

Agent Harness 概念的爆发标志着 AI 工程领域的一个关键拐点。大家终于认识到：当模型跨过能力阈值后，真正的工程挑战不在于让模型更聪明，而在于更有效地引导和约束模型的能力。

Harness 就是产品。

2026 年的 AI 工程竞赛不在模型层，因为各家公司的模型底座正在快速收敛；真正的竞赛在 Harness 层——谁能够设计出最优雅、最可靠、最可扩展的基础设施，来将模型的原始智能转化为可信赖的生产力。

下次你的 Agent 失败时，不要责怪模型，先去检查你的 Harness。谢谢大家。
